## 1. מטרה

להפוך את HANAN-APP ממתכנן פריסות גנרי לכלי מכירות של מתחם אירועים ספציפי: מעצבים את האירוע בעורך ה־2D/3D הקיים **בתוך מודל האולם האמיתי**, ובלחיצת כפתור מפיקים הדמיה פוטוריאליסטית של האירוע באולם באמצעות מודל תמונה AI.

**חלוקת אחריות (עקרון־העל ממסמך הרקע):**
- הדפדפן מספק **זהות ודיוק** — גאומטריה, קנה מידה, מיקומים, כמויות, צבעים, חומרים, מצלמה.
- ה־AI מספק **ריאליזם בלבד** — תאורה, השתקפויות, פירוט בדים, אינטגרציה צילומית.

## 2. סיכום הבנה (נעול מול המשתמש)

- **למי:** בעל אולם אחד ואיש המכירות שלו בלבד. מערכת פנימית, ללא הרשמות/חשבונות, לשימוש בפגישות מכירה.
- **חמש תצורות אולם:**
  1. אולם הריזורט (קומה א')
  2. שטח החוץ של הריזורט (בריכה / חצר)
  3. האולם הגדול (קומה ב') — מלא
  4. האולם הגדול מחולק — חצי אולם (שני החצאים זהים ⇒ חבילה אחת `grand-half` לשניהם)
  5. גן האירועים בקומת הקרקע
- **מודלי האולמות:** המשתמש בונה ב־SketchUp ומייצא GLB (רצוי עם טקסטורות מצילומי האולם האמיתיים). אין סריקות (Matterport/Gaussian Splatting ירדו מהשולחן).
- **ריהוט:** מ־Tripo עם סגמנטציה סמנטית לחלקים, דרך צינור הכנה (ניקוי → material slots → GLB) לפני כניסה לקטלוג.
- **מצלמות:** זוויות חתומות מראש בלבד — 3–5 לכל תצורה, מכוילות פעם אחת מול תמונת אמת מאותה נקודה.
- **מודל AI:** GPT Image 2 (OpenAI) — מעבד קלט ב־high input fidelity תמיד; מאחורי ממשק ספק־אגנוסטי.

**הכלל (הוכרע 2026-07-15): כל אלמנט שאינו קישוטי/עיצובי = קבוע במודל האולם.**

**אלמנטים קבועים (חלק מ־GLB האולם):** קירות, רצפה, תקרה/סבכת פלדה + גופי תאורה, בריכה, במה, רחבת ריקודים מרכזית, מעקה זכוכית + עמודי זהב, קירות מחוררים, מחיצות עץ, קיר וידאו, נוף חיצוני, **וכן בר, עמדת DJ, בופה, תחנות אוכל, אזורי ישיבה קבועים**.
**נגרר מהספרייה (Tripo):** **שולחנות** (6 סוגים) · **כיסאות** (6 סוגים) · **חופות ושערי טקס** · **בר משקאות** (נייד, חריג לכלל) · **קישוטים** — קישוטי שולחן (סנטרפיסים), קישוטי חופה, קישוטים תלויים מהתקרה. מפרט מלא: `furniture-library-spec.md`.

**קבוע הוא פר־אולם:** בריזורט החופה הקבועה הוסרה ב־2026-07-21 והוחלפה ב־`ZONE_HUPA`; חופות ושערי טקס נגררים מהספרייה וננעלים לאזור הזה. אולם תחתון: כולל את גן האירועים כקבוע (חלק מה־GLB), ובו בלבד אופציית שולחן עם/בלי מפה (למלבני+מרובע). כרגע קיים רק GLB הריזורט (ריק מריהוט); אולם תחתון + גן בהמשך.

**מטרות־אי (Non-goals):** לא פלטפורמה רב־אולמית; לא שירות עצמי ללקוחות; לא מצלמה חופשית להדמיות (בשלב זה); לא תקריבי שולחן (תוספת עתידית); לא רינדור צד־שרת.

## 3. הנחות

- זמן הפקה 15–45 שניות מקובל בפגישת מכירה; ההפקה אינה חוסמת עריכה.
- מקובל לשלוח את צילומי הסצנה ותמונות האולם לספק AI חיצוני (תמונות שיווקיות, לא מידע רגיש).
- עלות פר־הדמיה — סנטים; אין צורך במנגנון מכסות.
- המשתמש יספק תמונות אמת של כל אולם מכל זווית חתומה.
- להדמיות חוץ נדרשת בחירת שעת יום (יום / שקיעה / ערב).
- האפליקציה נשארת מקומית (IndexedDB, ללא שרת); מפתח ה־API נשמר מקומית והקריאה נעשית ישירות מהדפדפן. אם המערכת תיפתח החוצה — יתווסף proxy דק (נקודת ההחלפה היחידה).
- תוצר ההדמיה: PNG לשמירה/שיתוף/הדפסה.

## 5. חבילת אולם (Venue Pack)

תיקייה סטטית `public/venue-packs/<id>/`:

| קובץ | תוכן |
|---|---|
| `manifest.json` | id, שם בעברית, גרסת סכמה, סוג (indoor/outdoor) |
| `venue.glb` | המודל מ־SketchUp. מטרים ביצוא → ס"מ בטעינה. נקודת אפס = פינת האולם התכנונית (תואם ל־plan space). שמות צמתים מוסכמים: `ceiling` / `walls` / `floor` / `fixed_*` |
| `footprint.json` | גבול האולם (פוליגון, ס"מ) + קטעי קיר לסנאפ + אזורים חסומים (עמודים, במה קבועה, מדרגות) + פתחי דלתות |
| `cameras.json` | לכל זווית: id, תווית בעברית, מיקום+גובה+יעד (ס"מ), FOV אנכי, יחס תמונה, שם קובץ רפרנס, תבנית פרומפט |
| `refs/<angle>.jpg` | תמונת אמת שצולמה מאותה נקודה באולם |
| `lighting.json` | פנים: ריג ברירת מחדל · חוץ: יום/שקיעה/ערב — כל פריסט = הגדרות תאורת three.js + רמז פרומפט תואם |

**ולידציה:** סכמות zod לכל קבצי ה־JSON (התשתית כבר בפרויקט). חבילה פגומה ⇒ fallback לאולם פרוצדורלי + אזהרה ברורה; הפרויקט לעולם לא נתקע. גרסת חבילה עתידית ⇒ הודעה "עדכן את האפליקציה".

### תהליך הכנת חבילה (חד־פעמי לכל אולם, מחוץ לאפליקציה)

1. ייצוא GLB מ־SketchUp (עם טקסטורות מצילומי האולם).
2. **סקריפט הכנה** (Node, בריפו): חותך את ה־GLB בגובה ~1מ' → מפיק טיוטת `footprint.json` → תיקון ידני לפי הצורך. אם קיים DXF מאוטוקאד — משמש לאימות מידות חד־פעמי בלבד (לא פורמט ריצה — מקור אמת אחד לכל אולם).
3. צילום תמונות אמת מהזוויות המתוכננות.
4. **מצב כיול** (כלי מוסתר באפליקציה): רינדור מעל תמונת האמת בשקיפות 50% → מזיזים את המצלמה עד התלכדות קירות → "שמור" מעדכן את `cameras.json`. זהו הצעד שקובע את איכות כל ההדמיות.

הוספת אולם/זווית בעתיד = חזרה על התהליך; אפס שינויי קוד.

### 5.1 כלי המרה וייצוא GLB (הוכרע 2026-07-14)

| כלי | תפקיד | סטטוס |
|---|---|---|
| **SimLab MCP** | המרת SKP→GLB בפקודה אחת (`simlab_convert`) + בדיקת מבנה הסצנה (`simlab_inspect_scene`) + עריכה (שמות צמתים, חומרים). SimLab Composer מותקן וה־MCP כבר מחובר לסשן | **המסלול הראשי** — לא דורש התקנה נוספת |
| **סקריפט עזר (Node)** | ולידציה/נרמול על ה־GLB המיוצא: בדיקת קנה־מידה (bbox), y-up, קיום שמות צמתים חובה, מרכוז לפינת אפס. חלק מ"צינור הכנת הנכסים" של מסמך העקרונות | ייכתב אחרי בדיקת מבנה GLB אמיתי (לא ספקולטיבי) |
| **mhyrr/sketchup-mcp** (קהילתי) | שליטה חיה במודל SketchUp (rename layers, set origin, set material, `eval_ruby`) לפני ייצוא | אופציונלי — רק אם צריך לתקן מודל חי; דורש התקנת ‎.rbz + uv |
| ~~SketchUp Connector for Claude (רשמי)~~ | — | **לא מתאים**: יוצר ‎.skp חדשים בלבד, לא קורא/מייצא מודל קיים |

**נקודת אפס וכיוון:** האפס הוא הרצפה, והמשתמש כבר מיקם את המודל לנקודת האפס. **הכיוון ה"קדמי" קריטי פר־אולם** — בריזורט הקדמי פונה למדבר, והמדבר ברקע הוא חלק מזהות ההדמיה; יש לשמר את ה"hero facing" הזה ב־`cameras.json` וכל זוויות ההדמיה נגזרות ממנו.

### 5.2 ממצאי המרה בפועל — ריזורט גאמוס (2026-07-14)

המרת `ריסורט גאמוס.skp` → GLB דרך SimLab ובדיקת המבנה בפועל העלו:

| ממצא | ערך | משמעות לצינור ההכנה |
|---|---|---|
| **קנה מידה** | מטרים (bbox ≈ 84×33.5×13 מ') | ✅ כמצופה; האפליקציה תמיר מ׳→ס"מ. שים לב: ה־84מ' כולל את **גיאומטריית המדבר/הנוף החיצוני** — מצוין להדמיה, אבל צריך לגזור את ה־footprint מרצפת האולם בלבד |
| **ציר "מעלה"** | **Y-up** (גובה 13מ' על Y; רצפה ב־Y≈0) — *תוקן:* ה־GLB תקני Y-up; תצוגת SimLab (Z-up) הטעתה | ✅ אין צורך בסיבוב. הכלי מדפיס bbox לאימות; `--source-up z` שמור למקרה עתידי |
| **נקודת אפס** | bbox min ≈ [-8.5, -1.45, ...] (רצפה ~Y=0; גלישה שלילית = בריכה/נוף) | סומכים על מיקום המשתמש; מרכוז venue הוא opt-in (`--recenter`) |
| **משקל** | **59MB, 9077 צמתים → 3.6MB אחרי הכלי (94%)** ✅ | Draco+webp; **merge (flatten/join) כבוי** — ניפח bbox 13→28מ' על scale מקונן. draw-calls יטופלו ב־three אם יידרש |
| **שמות צמתים** | אוטומטיים (`Assembly-N`) — SimLab איבד את שמות השכבות מ־SketchUp | לא להסתמך על שמות צמתים לכיבוי/הדלקה ב-v1; לטעון את כל המעטפת. חלופה עתידית: בחירה לפי **חומר** (ראו למטה) |
| **חומרים** | ✅ נשמרו עם שמות אמת: `Marble_01_1K`, `Tile_Interior_14_1K`, `Wood_Veneer_15_1K`, `Glass Railing Color`, `LED Grid`, `[Charcoal]` ועוד | ✅ המודל **צבוע בחומרי אמת** — הזהות תעבור להדמיה; מאשר את החלטת "חומרי המודל = חומרי האמת" |
| **תוספות SimLab** | צמתי `Camera-1` ו-`Ground` | להסיר בהכנה |

**סקריפט העזר — נבנה ואומת ✅ (`tools/glb-prep/`, 2026-07-15):** כלי Node עצמאי מבוסס `@gltf-transform`, שני מצבים: `venue` (strip extras + dedup/prune/weld + webp + Draco) ו־`prop` (ריסקייל לממדים אמיתיים בס"מ + מרכוז + נרמול, batch על תיקייה). אומת על מודל הריזורט: ‏59MB→3.6MB, גיאומטריה תקינה וזקופה, 43 חומרים נשמרו (רנדר). תיעוד מלא: `tools/glb-prep/README.md`. **ל-Tripo:** `--mode prop --height 92` (כיסא) / `--diameter 180` (עגול) / `--footprint 240x120` (מלבני).

## 6. זרימת "הפקת הדמיה"

**UI:** כפתור "הפקת הדמיה ✨" בסרגל הכלים → דיאלוג: בחירת זווית (כרטיסיות עם תמונות refs), בחירת תאורה (חוץ בלבד), הערה חופשית אופציונלית → Generate. לא חוסם עריכה.

**מתחת למכסה:**
1. **צילום** — קפיצת מצלמה לזווית החתומה; פריים בודד off-screen ברזולוציית היעד (למשל 1536×1024), מצב "נקי" (בלי בחירה/גיזמו/רשת — מקביל ל־`clean` של captureBus הקיים); המצלמה חוזרת מיד.
2. **בקשה** — `prompt = תבנית הזווית + רמז תאורה + הערת משתמש`. ליבת התבנית: *"שמור בדיוק על מיקום, כמות וצבעי הריהוט מהתמונה הראשונה; החל את החומרים, התאורה והאווירה מתמונת הרפרנס"*. תמונות: [רינדור, תמונת אמת של הזווית].
3. **קריאה** — דרך הממשק האגנוסטי:

```ts
interface ImageProvider {
  generate(req: {
    images: Blob[]        // [render, refPhoto]
    prompt: string
    size: { width: number; height: number }
    signal: AbortSignal
  }): Promise<Blob>
}
```

מתאם GPT Image 2 מממש מול `/v1/images/edits`. החלפת ספק = מתאם חדש בלבד.

4. **Best-of-2 + QA אוטומטי** (ממצא PoC, ריצות 1–2): הבקשה מפיקה **שתי מועמדות** במקביל — נצפה בפועל שברוב המקרים לפחות אחת נאמנה 1:1. **מודל QA ויזואלי** (LLM ראייה, דרך אותו ממשק ספק־אגנוסטי) מקבל את הרינדור + כל מועמדת ומחזיר דו"ח מובנה: ספירת שולחנות תואמת? אשכולות במקומם? כיסאות מדגמיים? צבעים? — המועמדת שעוברת נבחרת אוטומטית. שתיהן נכשלו ⇒ ריצה חוזרת (עד 2 ניסיונות) ואז הצגת המועמדת הטובה עם אזהרת סטיות. הבדיקה קפדנית במיוחד במבטי־על, שבהם חובת התאמת 1:1 של אלמנטים.
5. **תוצאה** — מסך השוואה (רינדור מול הדמיה זו לצד זו + דו"ח ה־QA = בקרת נאמנות מיידית). פעולות: שמירה לגלריה / הפקה מחדש / הורדת PNG / שיתוף.

**גלריה:** כל הדמיה נשמרת ב־IndexedDB עם מטא־דאטה (זווית, תאריך, פרומפט, ספק) **ורינדור המקור שלה** — תמיד אפשר להשוות מה תוכנן מול מה שהוצג ללקוח. נגישה מהעורך ומכרטיס הפרויקט. הגלריה מציגה נפח תפוס ומאפשרת מחיקה; מחיקת פרויקט מוחקת את הגלריה שלו.

**שלב שני אופציונלי — תקריבים (בונוס, לפי בקשת לקוח):** מתוך הגלריה, על הדמיה שכבר נוצרה, פעולת "תקריב שולחן": ההדמיה הקיימת (לא הרינדור!) נשלחת שוב למודל עם פרומפט תקריב (עדשה 50mm, גובה ישיבה, עומק שדה רדוד, סטיילינג שולחן ערוך — מתבניות ה־PDF). כך התקריב יורש את הזהות, התאורה והאווירה של התמונה הראשית ונשאר עקבי איתה. התקריבים נשמרים בגלריה כצאצאים של תמונת המקור.

### 6.1 מדיניות אפסקייל (שני מסלולים)

האפסקייל הוא שלב אחרון אופציונלי להגדלה/הדפסה. שני מסלולים נפרדים לפי סוג התמונה, ושניהם רצים דרך ComfyUI (ענן) או מיקום שייקבע:

| סוג תמונה | מנוע אפסקייל | תיאור תמונה (prompt) |
|---|---|---|
| **הדמיה ראשית** (מבט־על/מבט־עין רחב) | SeedVR2 (אומת בריצה 1 של ה־PoC) | לא חובה — אפסקייל נאמן |
| **תקריב** (שולחן/כיסא/פרט) | **Magnific בלבד** | **חובה** — Magnific הוא upscaler יצירתי מונחה־פרומפט; תיאור התמונה מכוון איזה פרטים להזות (בדים, כלים, טקסטורות). בקרות Creativity/HDR/Resemblance נמוכות לשמירת נאמנות |

**למימוש:** תיאור התמונה לתקריב ייגזר אוטומטית מפרומפט התקריב + מטא־דאטה של הפריט (סוג שולחן, צבע מפה). המנוע (SeedVR2 / Magnific) הוא קונפיגורציה, לא ארכיטקטורה — מאחורי אותו רעיון ממשק אגנוסטי כמו ספק התמונה.

## 7. אכיפה קשיחה של גבולות (החלטת משתמש)

אובייקט **לא יכול** לצאת מגבול ה־footprint או להיכנס לאזור חסום:
- נקודת האכיפה: שכבת הפעולות (`state/actions.ts` — מסלול הכתיבה היחיד) ⇒ תופסת גרירה, הדבקה, שכפול, חיצי מקלדת ושינוי גודל במקום אחד.
- במהלך גרירה: הצמדה לגבול (clamp). הנחה לא־חוקית: החזרה למקום + היזון חוזר אדום.

## 8. טיפול בשגיאות

| תרחיש | התנהגות |
|---|---|
| אין מפתח / מפתח שגוי | נבדק מראש; הודעה בעברית + קיצור להגדרות |
| אין רשת | האפליקציה עובדת כרגיל (מקומית); רק כפתור ההדמיה מנוטרל עם הסבר; "נסה שוב" בלי לאבד הגדרות |
| Rate limit / עומס ספק | ניסיון חוזר אוטומטי אחד עם השהיה → הודעה ידידותית |
| ביטול (סגירת דיאלוג) | AbortController — הבקשה מבוטלת בפועל |
| אובדן WebGL context | try/recover: יצירת renderer מחדש + ניסיון שני לפני שגיאה |
| GLB/חבילה פגומים | ולידציית zod; fallback פרוצדורלי + אזהרה |
| ה־AI המציא/השמיט ריהוט | לא נתפס בקוד; ההגנה: מסך ההשוואה + הפקה מחדש; מבטים רחבים = סיכון מוכר ומתועד בתבנית הזווית |

## 9. בדיקות ו־PoC

**שלב 0 — שער Go/No-Go (לפני כל קוד):** רינדור SketchUp של אולם + תמונת אמת → GPT Image 2 עם הפרומפט מהעיצוב, על 3–4 זוויות. בודקים שימור כמות/מיקום/צבע ריהוט וזהות אולם. נאמנות לא מספקת גם אחרי כיול פרומפט ⇒ הסלמה לגישה ב' (רב־אותות: depth+segmentation דרך ControlNet בשרת) לפני השקעה בתשתית.

**בדיקות יחידה (vitest):** גאומטריית footprint (נקודה־בפוליגון, clamp, סנאפ קירות, אזורים חסומים); ולידציית manifest + מיגרציית venuePackId; מודול render (הרכבת פרומפט, בניית בקשה עם fetch מדומה, שגיאות וביטול).

**אינטגרציה ידנית־ויזואלית:** טעינת כל חבילה (GLB מוצג, footprint מתלכד עם הרצפה); מצב הכיול כאימות מצלמות; זרימת Generate מקצה לקצה בדפדפן אמיתי. כלל פרויקט: אין "בוצע" בלי אימות ויזואלי.

**בקרת נאמנות שוטפת:** צ'קליסט השוואה (כמויות/צבעים/מיקומים) מול רינדור המקור לפני הצגה ללקוח.
